# Отчет по лабораторной работе №5

## 1. Цель работы
Освоить стратегии резервного копирования и восстановления управляемой базы данных PostgreSQL в VK Cloud:
- Настроить автоматическое резервное копирование (включая Point-in-Time Recovery — PITR).
- Создать и восстановить базу данных из резервной копии.
- Организовать хранение бэкапов в объектном хранилище S3.
- Отработать сценарий восстановления после сбоя (Disaster Recovery).

Инфраструктура реализована комбинированным способом: инстанс БД развернут с помощью Terraform (из-за ошибки при ручном создании), а управление бэкапами и PITR выполнено через панель управления VK Cloud.

## 2. Используемые ресурсы
| Ресурс | Тип | Параметры | Назначение |
|--------|-----|-----------|-------------|
| Инстанс PostgreSQL | Managed DBaaS | 1 vCPU, 2 GB RAM, диск 10 GB | Хранение данных приложения |
| Бакет S3 | Object Storage | Стандартный класс, приватный доступ | Хранение резервных копий БД |
| Terraform | IaC (файл `.tf`) | Провайдер vkcs | Создание инстанса БД |
| Автоматические бэкапы | Managed Backup | Retention 7 дней (по заданию) | Ежедневное резервное копирование |
| PITR | Point-in-Time Recovery | Окно 24 часа (максимум) | Восстановление на произвольный момент времени |

## 3. Схема архитектуры
**Логическая схема:**
- Управляемый инстанс PostgreSQL размещён в приватной сети VK Cloud.
- Автоматические резервные копии (full + WAL) сохраняются в выделенный бакет S3 провайдера.
- Настроена политика хранения бэкапов (retention) и возможность PITR.
- Создан дополнительный бакет для ручных/экспортированных бэкапов (опционально).
- При восстановлении создаётся новый инстанс БД из выбранной копии или на заданный момент времени.

## 4. Ход выполнения

### Этап 1. Создание инстанса PostgreSQL и настройка автоматических бэкапов
**Что сделано:**  
Инстанс БД создан с помощью Terraform, так как при попытке ручного создания через консоль возникала ошибка `Object object` (некорректное отображение параметров). Terraform-конфигурация `VKC_LR_05_DB_РемизовКЛ_20260610.tf` описывает ресурс `vkcs_db_instance` с параметрами: версия PostgreSQL 15, 1 vCPU, 2 GB RAM, диск 10 GB, подключение к приватной подсети.

**Ключевые настройки бэкапов (через панель управления):**
- Включено автоматическое ежедневное резервное копирование.
- Задан retention (срок хранения) — 7 дней (согласно заданию). Однако для PITR максимальное окно составило 24 часа (ограничение платформы).
- Бэкапы автоматически сохраняются в служебный бакет S3 провайдера.
- Дополнительно создан пользовательский бакет для возможного экспорта ручных дампов.

**Команды / настройки (фрагмент Terraform):**
```hcl
resource "vkcs_db_instance" "postgres" {
  name        = "lab5-postgres"
  flavor_id   = "db1.small"
  datastore {
    version = "15"
    type    = "postgresql"
  }
  volume {
    size = 10
    type = "ceph-ssd"
  }
  network {
    uuid = vkcs_networking_network.main.id
  }
}
```
**Результат:**  
Инстанс успешно создан, автоматические бэкапы начали формироваться. Скриншот списка бэкапов показывает наличие ежедневных копий.

### Этап 2. Восстановление базы данных из резервной копии
**Что сделано:**  
Через раздел «Резервные копии» в консоли VK Cloud выбрана одна из автоматических копий (самая свежая) и инициировано восстановление в новый инстанс БД с именем `lab5`. Процесс занял около 5 минут.

**Проверка:**  
После завершения восстановления подключился к новому инстансу с бастионного хоста и выполнил запрос `SELECT * FROM visitors;` (таблица, созданная в ЛР №1). Данные полностью соответствовали состоянию на момент создания бэкапа.

**Результат:**  
Новый инстанс БД успешно запущен, все данные сохранены. Скриншот `VKC_LR_05_BackupDB_РемизовКЛ_20260610.png` подтверждает процесс восстановления.

### Этап 3. Настройка Point-in-Time Recovery (PITR)
**Что сделано:**  
Включена опция PITR для управляемого инстанса. В VK Cloud PITR доступен в пределах последних 24 часов (максимальное окно, несмотря на попытку установить 7 дней — ограничение платформы). Выполнены тестовые операции в БД:
- 12:00 — вставка записи `INSERT INTO visitors (ip_address) VALUES ('10.0.1.100');`
- 12:05 — удаление этой записи `DELETE FROM visitors WHERE ip_address = '10.0.1.100';`

Затем через консоль выбрана точка восстановления на момент времени 12:02 (до удаления). Запущен процесс PITR-восстановления в новый инстанс.

**Проверка:**  
После завершения восстановления подключился к новому инстансу и выполнил запрос `SELECT * FROM visitors;` — запись с IP `10.0.1.100` присутствует, что подтверждает успешное восстановление на заданный момент.

**Результат:**  
PITR работает корректно, но с ограничением окна в 24 часа (вместо желаемых 7 дней). Скриншот `VKC_LR_05_PITR_РемизовКЛ_20260610.png` фиксирует настройки.

## 8. Ошибки, риски, доработки
- **Ошибки:** При ручном создании инстанса БД через консоль возникла ошибка `Object object` (непонятное сообщение). Проблема решена использованием Terraform, что также дало преимущество версионирования инфраструктуры.
- **Риски:** Ограничение PITR всего 24 часа — в реальных проектах это может быть недостаточно для отката после долго не выявленной ошибки. Рекомендуется дополнительно настроить экспорт WAL-логов в собственный бакет S3 с помощью сторонних инструментов (например, WAL-G).
- **Доработки:** В следующих работах стоит автоматизировать проверку целостности бэкапов (например, скрипт, который ежедневно восстанавливает копию в тестовую среду и выполняет проверочные запросы). Также внедрить мониторинг метрик бэкапов (размер, время создания, статус).

## 9. Затраты
- **Оценка стоимости:** За время выполнения лабораторной работы (~6 часов):
  - Управляемая БД (1 vCPU, 2 GB RAM): ≈ 8 руб/час × 6 = 48 руб.
  - Бакет S3 (хранение бэкапов, ≈ 5 ГБ): ≈ 0,5 руб/час × 6 = 3 руб.
  - Сетевые ресурсы, Terraform-стейт (минимально): ≈ 1 руб/час × 6 = 6 руб.
  - Прочие сервисы (например, бастион для подключения): ≈ 27 руб.
  - **Итого:** ≈ 84 руб. (совпадает с оценкой в metadata).
- **Время жизни ресурсов:** ~6 часов (с момента создания инстанса до удаления).
- **Все ли ресурсы удалены:** Да. Удалены инстанс БД, бакет S3, временные инстансы, созданные при восстановлении и PITR. В биллинге активных начислений нет.
