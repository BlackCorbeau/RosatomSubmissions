# Отчет по лабораторной работе

## 1. Цель работы
Спроектировать и развернуть отказоустойчивую архитектуру для веб-приложения, включающую:
- Изолированную сеть (VPC) с публичной и приватной подсетями.
- Два веб-сервера в приватной подсети.
- Балансировщик нагрузки в публичной подсети.
- Управляемую базу данных PostgreSQL в приватной подсети.
- Бастионный хост для безопасного доступа к внутренним ресурсам.

Инфраструктура реализована комбинированным способом: сеть, группы безопасности и балансировщик описаны в Terraform; виртуальные машины и база данных созданы вручную через интерфейс VK Cloud.

## 4. Ход выполнения

### Шаг 1. Создание VPC и подсетей (Terraform)
**Что сделано:**  
Написан Terraform-манифест `network.tf`, который создаёт VPC `lab1-vpc`, публичную подсеть `lab1-public` (10.0.1.0/24) и приватную подсеть `lab1-private` (10.0.2.0/24). Подсети подключены к существующему роутеру, что обеспечивает доступ в интернет для публичной подсети и NAT-маршрутизацию для приватной.

**Команды / настройки:**
```hcl
# network.tf
data "vkcs_networking_router" "existing" {
  id = "913bdd47-9155-45b2-b404-5c10d0086132"
}

resource "vkcs_networking_network" "lab1_vpc" {
  name           = "lab1-vpc"
  admin_state_up = true
}

resource "vkcs_networking_subnet" "public" {
  name            = "lab1-public"
  network_id      = vkcs_networking_network.lab1_vpc.id
  cidr            = "10.0.1.0/24"
  dns_nameservers = ["8.8.8.8", "8.8.4.4"]
}

resource "vkcs_networking_subnet" "private" {
  name            = "lab1-private"
  network_id      = vkcs_networking_network.lab1_vpc.id
  cidr            = "10.0.2.0/24"
  dns_nameservers = ["8.8.8.8", "8.8.4.4"]
}

resource "vkcs_networking_router_interface" "public" {
  router_id = data.vkcs_networking_router.existing.id
  subnet_id = vkcs_networking_subnet.public.id
}

resource "vkcs_networking_router_interface" "private" {
  router_id = data.vkcs_networking_router.existing.id
  subnet_id = vkcs_networking_subnet.private.id
}
```
**Результат:**  
VPC и обе подсети созданы, публичная подсеть имеет доступ в интернет через роутер, приватная — только исходящий через NAT (без входящего из интернета).

### Шаг 2. Настройка безопасности (Terraform + ручное создание бастиона)
**Что сделано:**  
- Написаны Security Groups в `security.tf`:  
  - `bastion-sg`: разрешён SSH только с моего IP (95.37.52.68/32).  
  - `web-sg`: разрешён SSH только с `bastion-sg`, HTTP только с `lb-sg`.  
  - `lb-sg`: разрешён HTTP из интернета (0.0.0.0/0).  
- Бастионный хост создан вручную в публичной подсети с привязкой `bastion-sg` и ключом SSH.

**Команды / настройки:**
```hcl
# security.tf (фрагмент)
resource "vkcs_networking_secgroup" "bastion_sg" {
  name        = "bastion-sg"
  description = "SSH from my IP"
}
resource "vkcs_networking_secgroup_rule" "bastion_ssh" {
  direction         = "ingress"
  protocol          = "tcp"
  port_range_min    = 22
  port_range_max    = 22
  remote_ip_prefix  = var.my_ip   # 95.37.52.68/32
  security_group_id = vkcs_networking_secgroup.bastion_sg.id
}
# Аналогично для web_sg и lb_sg
```
**Результат:**  
Бастион доступен по SSH с моего IP. Веб-серверы и БД недоступны напрямую из интернета, доступ к ним возможен только через бастион.

### Шаг 3. Развертывание веб-серверов (ручное через консоль VK Cloud)
**Что сделано:**  
В приватной подсети созданы две ВМ `web-1` и `web-2` (Ubuntu 22.04, 1vCPU, 1GB RAM). При создании указан user-data скрипт для установки и настройки NGINX с индивидуальной страницей, отображающей hostname.

**Команды / настройки (user-data):**
```bash
#!/bin/bash
apt update
apt install -y nginx
echo "<!DOCTYPE html>
<html>
<head><title>Web Server $(hostname)</title></head>
<body>
    <h1>Welcome to $(hostname)</h1>
    <p>Server IP: $(hostname -I | awk '{print $1}')</p>
    <p>Backend server</p>
</body>
</html>" > /var/www/html/index.html
systemctl enable nginx
systemctl start nginx
```
**Результат:**  
Обе ВМ активны, NGINX работает. С бастиона выполнены `curl http://10.0.2.5` и `curl http://10.0.2.6` – возвращаются страницы с именами `web-1` и `web-2`.

### Шаг 4. Настройка балансировщика нагрузки (Terraform + ручное добавление членов)
**Что сделано:**  
- Terraform создаёт балансировщик `lab1-lb` в публичной подсети, HTTP-слушатель на порту 80, целевую группу `web-targets` с алгоритмом ROUND_ROBIN и health monitor (HTTP, `/`, интервал 10 с).  
- Веб-серверы добавлены в целевую группу вручную через интерфейс VK Cloud (или через OpenStack CLI), так как ресурс `vkcs_lb_member` не был включён в Terraform.

**Команды / настройки:**
```hcl
# loadbalancer.tf
resource "vkcs_lb_loadbalancer" "main" {
  name          = "lab1-lb"
  vip_subnet_id = vkcs_networking_subnet.public.id
}
resource "vkcs_lb_listener" "http" {
  name            = "http-listener"
  protocol        = "HTTP"
  protocol_port   = 80
  loadbalancer_id = vkcs_lb_loadbalancer.main.id
}
resource "vkcs_lb_pool" "web" {
  name        = "web-targets"
  protocol    = "HTTP"
  lb_method   = "ROUND_ROBIN"
  listener_id = vkcs_lb_listener.http.id
}
resource "vkcs_lb_monitor" "web" {
  name        = "web-monitor"
  type        = "HTTP"
  delay       = 10
  timeout     = 5
  max_retries = 3
  url_path    = "/"
  pool_id     = vkcs_lb_pool.web.id
}
```
**Результат:**  
Балансировщик получил публичный IP. В целевую группу добавлены оба веб-сервера, их статус – Healthy. При открытии публичного IP в браузере страницы чередуются между `web-1` и `web-2`.

### Шаг 5. Развертывание базы данных (ручное через консоль VK Cloud)
**Что сделано:**  
Создан управляемый инстанс PostgreSQL 15 (1 vCPU, 2GB RAM) в приватной подсети. Созданы пользователь `webapp` и база данных `webapp_db`. С бастиона выполнено подключение, создана тестовая таблица `visitors` и вставлена запись.

**Команды / настройки:**
```sql
CREATE TABLE visitors (
    id SERIAL PRIMARY KEY,
    ip_address INET,
    visit_time TIMESTAMP DEFAULT NOW(),
    user_agent TEXT
);
INSERT INTO visitors (ip_address) VALUES ('10.0.1.10'::inet);
SELECT * FROM visitors;
```
**Результат:**  
Подключение с бастиона успешно, таблица создана, данные вставлены. БД изолирована в приватной сети.

### Шаг 6. Очистка ресурсов
**Что сделано:**  
Ресурсы удалены в обратном порядке: балансировщик → веб-серверы → бастион → инстанс БД → Security Groups → VPC и подсети.

## 6. Ответы на вопросы задания

1. **Вопрос:** Какие IP-адреса зарезервированы в подсети и для чего?  
   **Ответ:** В подсети с CIDR /24 зарезервированы: первый адрес (10.0.x.0) – адрес сети; второй (10.0.x.1) – шлюз по умолчанию (роутер); третий (10.0.x.2) – DNS/DHCP сервер. Остальные адреса доступны для ВМ.

2. **Вопрос:** Почему приватная подсеть не имеет прямого доступа из интернета?  
   **Ответ:** Приватная подсеть не подключена к интернет-шлюзу напрямую, а все входящие соединения блокируются на уровне роутера и правил security groups, что изолирует внутренние ресурсы.

3. **Вопрос:** Почему SSH на веб-серверы разрешен только с бастиона?  
   **Ответ:** Это реализует схему jump host – все административные подключения идут через единственную защищённую точку входа, уменьшая атакующую поверхность.

4. **Вопрос:** Какое правило разрешает веб-серверу обновлять систему?  
   **Ответ:** Исходящий трафик (egress) по умолчанию разрешён полностью в security groups, поэтому веб-сервер может обращаться к репозиториям пакетов.

5. **Вопрос:** Как проверить, что nginx запущен, не подключаясь к ВМ?  
   **Ответ:** Через health check балансировщика, через систему мониторинга (например, Prometheus) или через API VK Cloud, если оно поддерживает проверку состояния сервиса.

6. **Вопрос:** Что произойдет с веб-сервером после перезагрузки ВМ?  
   **Ответ:** Так как использован `systemctl enable nginx`, сервис автоматически запустится при старте, и страница останется доступной.

7. **Вопрос:** Какой алгоритм балансировки используется по умолчанию?  
   **Ответ:** В конфигурации указан `ROUND_ROBIN` – циклическое переключение между серверами.

8. **Вопрос:** Что произойдет, если одна из ВМ перестанет отвечать на health check?  
   **Ответ:** Балансировщик перестанет направлять трафик на эту ВМ до тех пор, пока она снова не начнёт проходить проверку. Это обеспечивает отказоустойчивость.

9. **Вопрос:** Почему БД размещена в приватной подсети, а не в публичной?  
   **Ответ:** Для безопасности – база данных не должна быть напрямую доступна из интернета, чтобы снизить риск атак (SQL-инъекции, перебор паролей). Доступ возможен только из приватной сети, где находятся веб-серверы и бастион.

10. **Вопрос:** Какие преимущества дает использование управляемой БД вместо установки на ВМ?  
    **Ответ:** Автоматическое резервное копирование, мониторинг, патчи безопасности, масштабирование, высокая доступность (при выборе конфигурации с репликами), отсутствие необходимости в ручном администрировании.

## 8. Ошибки, риски, доработки
- **Ошибки:** Не было ошибок при выполнении. Однако при использовании Terraform пришлось вручную добавлять члены целевой группы, так как ресурс `vkcs_lb_member` не был описан – это осознанное упрощение для демонстрации гибридного подхода.

- **Доработки:** В следующих лабораторных работах стоит полностью описывать все ресурсы в Terraform (включая членов LB) и использовать Packer для кастомизированных образов, что ускорит развёртывание.
